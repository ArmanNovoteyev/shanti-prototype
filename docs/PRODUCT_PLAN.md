## 🎯 ПРИНЦИПЫ ПЕРВОГО КЛИЕНТА

1. **Лояльно, но правильно.** Не демпинговать, не обещать лишнего, не
   работать в убыток — но и не драть по максимуму. Цель — **референс-
   кейс**, который потом покажем другим клиентам. Провал здесь = провал
   портфолио.
2. **Всё в письменном виде.** Даже если доверие высокое — договор,
   меморандум, ТЗ. «Джентльменские соглашения» с бизнесом — корень
   будущих конфликтов.
3. **Честные сроки × 1.75.** Если думаем «4 месяца» — говорим «7
   месяцев». Лучше опередить план чем опоздать. (v1.0 было ×1.5 —
   поднимаем, Арман один и учится на ходу.)
4. **Не обещаем сроки на первой встрече.** Возвращаемся с цифрами
   через 3-5 дней подумав.
5. **Арман один = bottleneck.** Закладываем перерывы, выходные,
   болезни. Не играем в супергероя.
6. **Каждый этап — демо Алине + владелице.** Не копим сюрприз на 7
   месяцев, показываем прогресс **ежемесячно + живой PWA через 2-3
   месяца**.
7. **Данные клиентов — святое.** 94-ФЗ РК, согласия, бэкапы. Утечка
   = конец бизнеса салона.
8. **Выход возможен с любой стороны.** Предусматриваем в договоре.
9. **Движение важнее совершенства.** Фаза 0 = 2 недели, не больше.
   Юрист не готов → идём с меморандумом. Developer-аккаунт ждёт
   верификацию → начинаем бэкенд параллельно. Не застреваем в
   подготовке. (Добавлено v1.1.)

## 🏗 АРХИТЕКТУРНЫЙ СТЕК (v1.2 — решение от 2026-04-23)

### Почему не Supabase

Первоначально план предполагал Supabase. **Пересмотрено в v1.2** по
доводам Армана:

1. **Раскрученный салон = быстрый рост.** Перешагнём 1000 активных
   клиентов в первые 3-6 месяцев. Supabase Pro $25/мес + доп.плата за
   bandwidth/storage на дистанции ощутимо.
2. **Vendor lock-in.** Для первого серьёзного продукта не хочется
   зависимости от одного вендора — особенно если через год-два планируется
   v2.0 как SaaS для спа-салонов.
3. **Экономия накопительно.** Свой VPS ~3500 тг/мес vs Supabase+Vercel
   ~23000 тг/мес = **экономия ~25 000 тг/мес = 300 000 тг/год**.
   За 5-6 лет окупает стоимость MVP.

### Финальный стек v1.2

**Backend (apps/api):**
- **Node.js 20 + Fastify + TypeScript** — быстрее Express, native TS
- **Drizzle ORM** — TypeScript-first миграции и запросы, без тяжёлого Prisma
- **Lucia Auth** (+ oslo) — SMS OTP авторизация, без vendor lock-in
- **WebSocket** (Fastify plugin) — live-уведомления админки о новых бронях
- **Argon2** — хеши
- **Zod** — валидация DTO (shared types между backend и frontend)

**Frontend (apps/client + apps/admin):**
- **React 18 + Vite 5 + TypeScript** — с первого дня `strict: true`
- **Client:** существующий V3-код переводится на TS, остаётся inline styles
- **Admin:** новый проект, можно TailwindCSS (admin с таблицами/формами — Tailwind ускоряет)
- **lucide-react** (иконки), **qrcode.react** (сертификаты)
- **date-fns** (даты)

**Mobile (apps/mobile):**
- **Capacitor** v6 — обёртка над client-bundle для iOS/Android (Фаза 4)
- Native: `@capacitor/push-notifications`, `@capacitor-community/barcode-scanner`

**Инфраструктура:**
```
┌─ VPS Hetzner CX22 (Frankfurt) — €4.50/мес ≈ 2 400 тг ─┐
│  Docker Compose + Traefik (TLS auto via Let's Encrypt) │
│  ├── Postgres 16        (наша БД)                      │
│  ├── apps/api (Fastify) (наш бэкенд)                   │
│  ├── Redis              (кеш, rate limit)              │
│  └── MinIO              (фото/сертификаты, S3-совместим)│
└────────────────────────────────────────────────────────┘

+ Backblaze B2       (бэкапы)         ~$1/мес ≈ 500 тг
+ Vercel free        (client + admin) 0 тг
+ Mobizon/SMSC       (SMS)            5-15к тг/мес по трафику
+ Kaspi Pay API      (платежи)        per-transaction комиссия
+ Sentry free        (мониторинг)     0 тг
+ Plausible/Umami    (аналитика)      $9/мес или self-host бесплатно
```

**Итого инфраструктура: ~8 000 тг/мес** (против 33 000 на Supabase+Vercel Pro).

### Монорепо vs два репо — МОНОРЕПО

**pnpm workspaces + Turborepo:**

```
shanti/
├── apps/
│   ├── client/              — PWA клиент (Vite + React + TS)
│   ├── admin/               — админка (Vite + React + TS)
│   ├── api/                 — бэкенд (Fastify + Drizzle + TS)
│   └── mobile/              — Capacitor обёртка v1.0
├── packages/
│   ├── db/                  — Drizzle схема + миграции + auto-generated types
│   ├── shared-types/        — DTOs, Zod-схемы, константы (Service, Booking)
│   ├── shared-ui/           — общие компоненты (Button, Card, Input) + design tokens
│   ├── shared-config/       — цвета, шрифты (Evolventa), тайминги
│   └── shared-utils/        — формат телефона/цены, i18n-хук
├── turbo.json
├── pnpm-workspace.yaml
└── package.json
```

**Что даёт:**
- Изменил схему БД → типы обновились в client, admin, api одним `pnpm db:generate`
- `turbo dev` поднимает все приложения параллельно
- CI бежит только на изменённых пакетах (Turbo cache)
- Общие React-компоненты из `shared-ui` переиспользуются — не копируем Button из client в admin
- Дизайн-токены из `shared-config` — один источник правды для всех трёх приложений

**Цена:** 1 день на настройку монорепо + привычка `pnpm` вместо `npm`.

### TypeScript с первого дня

- **Новый API:** Fastify + TS + strict mode
- **Новая админка:** React + Vite + TS + strict mode
- **Существующий V3-клиент:** `.jsx` → `.tsx` плавно по мере касания
  (Vite ест оба, не ломает). Новые файлы — сразу TS.
- **Shared types** из `packages/db/schema.ts` → автогенерация TypeScript-типов
  через Drizzle. Backend и frontend используют одни и те же типы.

### Серверная безопасность (замена Supabase RLS)

Без Supabase RLS защита прав доступа делается на уровне Fastify middleware:

- **Authorization header** → Lucia проверяет session cookie → пишет `req.user`
- **Role-based guards** на маршрутах: `@Auth('admin' | 'alina' | 'master' | 'owner')`
- **Query-level filters**: клиент видит только свои брони через `WHERE client_id = req.user.id`
- **Audit-log middleware** пишет в `audit_log` все изменения в критических таблицах
- **Rate limit** через `@fastify/rate-limit` + Redis
- **HTTPS** автоматически через Traefik + Let's Encrypt
- **Секреты** в `.env` на сервере, никогда в клиентском bundle

### GSD workflow — реальный speedup 1.5-2x (не 5x)

Используем существующий скилл **GSD с агентами** для структурирования работы:

**Где GSD реально ускоряет:**
- Структурированные фазы (`/gsd-discuss-phase` → `/gsd-plan-phase` → `/gsd-execute-phase` → `/gsd-verify-work`) — меньше переделок
- Параллельные агенты через `gsd-workstreams` — API + admin-UI одновременно, не последовательно
- `gsd-debug` с персистентным стейтом в `.planning/` — критично для 7-8 месяцев проекта, контекст не теряется между сессиями
- `gsd-code-review` + `gsd-secure-phase` + `gsd-ui-review` — ловят баги **до** того как Алина их увидит
- Атомарные коммиты с manifest фазы = **трассируемость платежей по траншам** (20% за Фазу 1 — вот hash, вот артефакты, вот что сделано)
- `gsd-verify-work` — goal-backward проверка что построили то что в ТЗ

**Где GSD НЕ ускоряет** (эти недели не сжимаются никаким скиллом):
- Apple ревью — недели
- Kaspi Pay юридические документы и интеграция — недели
- Юрист для договора — недели
- Ответы от Алины/владелицы — 3 рабочих дня по договору
- Тестирование у реальных пользователей v0.9 — недели
- Apple Developer верификация для РК — 2-4 недели

**Реальный расчёт:**

| Фаза | Без GSD | С GSD |
|------|---------|-------|
| Чистое программирование (Фазы 1-4) | ~16 нед | ~10-12 нед |
| Внешние процессы (Apple/Kaspi/юрист/фидбек) | ~10 нед | ~10 нед (не ускоряется) |
| Подготовка/планирование | ~3 нед | ~2 нед |
| **Итого** | **~29 нед** | **~24 нед** |

**Speedup ≈ 1.2x календарно, 1.5-1.7x по чистому коду.** Не 5x, но и не ноль.

**Главная ценность GSD сверх скорости:**
1. **Качество** — меньше багов на проде, меньше стресса с первым клиентом
2. **Traceability для договора** — каждая фаза с manifest, hash, артефактами
3. **Bus-factor** — если теряем контекст, `.planning/` восстанавливает всё
4. **Параллельность** нативна в GSD-агентах

### 2.1. Текущее состояние

### 2.2. Путь к настоящему приложению

**Шаг A. Заменить прототип-URL на продакшн-домен (Фаза 0)**

**Шаг B. Превратить в PWA (v0.9)**
- Добавить `manifest.json` с иконками
- Service Worker для базового кэширования (не полный офлайн пока)
- `<meta>` теги для iOS (apple-touch-icon, apple-mobile-web-app-capable)
- Клиент открывает URL → «Добавить на рабочий стол» → иконка

  пользоваться через PWA (через 2-3 месяца от старта)

**Шаг C. Обернуть в Capacitor для сторов (v1.0)**
- Наш React-код не меняется
- Capacitor добавляет iOS-проект (Xcode) и Android-проект (Android
  Studio)
- Native фичи: push, камера (для QR), геолокация
- Билд в сторы
- **OTA-стратегия:** web-часть обновляется без ревью через manual bundle
  replace; native-часть (пуши, камера) — через обновление в сторе

### 5.2. Резервное копирование

- **Ежедневный `pg_dump`** через cron на VPS → загрузка в Backblaze B2 (шифрование через age)
- **MinIO фото** — бэкап раз в неделю (фото меняются редко)
- **Retention policy**: ежедневные бэкапы хранятся 30 дней, еженедельные — 12 месяцев

- **Тестируем восстановление** раз в квартал (реальный drill)
  — **в Google Calendar как повторяющееся событие**, иначе забудется
- **Критично:** ключ шифрования бэкапов — в 1Password с доступом у Алины тоже

### 5.3. Аналитика

**Privacy-friendly (рекомендую):**

- Какие филиалы выбирают
- Конверсия сертификатов

**Sentry** — отлов runtime ошибок, free tier обычно хватает.

### 5.4. Безопасность

- **HTTPS везде** (Vercel для frontend + Traefik/Let's Encrypt на VPS для API)
- **Секреты в `.env` на VPS**, НЕ в коде (sanity check перед каждым коммитом, `.env` в `.gitignore`)
- **Role-based middleware guards** на Fastify — клиент видит только свои данные, мастер — только своих клиентов (замена Supabase RLS)
- **Rate limiting** через `@fastify/rate-limit` + Redis (защита от брутфорса SMS)
- **fail2ban на VPS** — автобан IP с большим количеством failed SSH/HTTP
- **Firewall (ufw)** — открыты только 22 (SSH), 80, 443 (HTTPS)
- **SSH только по ключам** (password auth отключён)
- **Регулярный аудит зависимостей** (`pnpm audit`) — раз в месяц
- **Обновления системы** (apt/docker) — раз в 2-3 месяца (входит в поддержку A 4.3)
- **Argon2 для хешей паролей админки** (если потребуется, клиенты через OTP)
